## Key Findings

1. The rogue homepage kept the right broad modules, but not the right visual language. Current live Polymarket still has a homepage hero, breaking-news module, hot-topics module, and an all-markets feed. The local version is materially darker, denser, and more dashboard-like than the live product.
2. Theme baseline is wrong. A direct browser check on 2026-04-19 with forced `light` and `dark` color schemes showed that live Polymarket follows system `prefers-color-scheme` by default. The local app hard-codes a dark shell. For this clone, baseline parity means system-pref-driven theming rather than a forced single theme.
3. Politics, crypto, and sports diverged harder than the homepage. Current live category pages are list-first and much lighter. The landed local pages read like custom internal dashboards rather than Polymarket category surfaces.
4. The current sports route is functionally broken. On 2026-04-19, a live top-volume sports slice fetched from Gamma returned `25` visible events at `limit=30`, and the first `12` were all futures/outrights. The local route defaults to a `Live` tab built from game detection, so the initial view renders an empty state instead of markets.
5. The current crypto filters are not grounded in current live data. On the same top-30 crypto slice, the rogue helpers resolved time buckets to only `all: 13` and `pre-market: 17`, market type to `all: 30`, and lead-coin chips to `all: 24`, `bitcoin: 4`, `ethereum: 1`, `microstrategy: 1`. Most of the shipped chip rows are therefore decorative or dead.
6. `EventGrid` should not have been moved behind a client boundary. Session `06` intentionally kept it server-side so only `EventCard` and `PriceCell` client islands pay the hydration cost. The current `'use client'` on `EventGrid` buys nothing and makes the bundle boundary worse.
7. The current tag contract is mostly correct and should stay narrow. `isEventVisible()` hiding only `hide-from-new` remains the right event-suppression rule; category headline tags still arrive as `forceHide: true` and should not suppress whole events.

## Architectural Critique

1. Client boundaries became too coarse.
   Current route entrypoints for the homepage, crypto, and sports all hand full event arrays to top-level client components (`HomePage`, `CryptoExplorer`, `SportsHub`), and `EventGrid` was also moved client-side. Architecturally, that means large deterministic trees are serialized and hydrated even though only a small subset of the UI is actually interactive. This is a poor trade for recruiter review because the app stops looking intentional about server/client boundaries.
2. Deterministic view derivation drifted into the client.
   Featured selection, breaking-news selection, topic aggregation, sports grouping, and crypto filter counts are all derivable from server-fetched data before the page is sent. Keeping these derivations inside large client components couples render logic to browser state, increases hydration work, and makes the eventual realtime layer harder to reason about because "static layout shaping" and "live price updates" are no longer clearly separated.
3. Category-route architecture forked prematurely.
   Before the data model was re-validated, `/crypto` and `/sports` moved from a shared server-filtered category baseline into bespoke client route entrypoints. That creates two architectural paths for category pages: one generic (`CategoryPage`) and two custom ones with different fetch caps, filter state, and grouping rules. That is manageable only if the specialized routes are unquestionably correct, which they are not today.
4. Data-contract assumptions were encoded too early.
   The rogue crypto and sports implementations inferred page structure from tag and title heuristics before checking whether the current Gamma slices actually support those assumptions. Good architecture here means isolating these in explicit facet/parsing helpers with strong fallbacks and tests, not letting unsupported assumptions drive the route entrypoint.
5. Realtime pressure was not treated as a first-class downstream constraint.
   Sessions `09` and `10` will subscribe one token atom per visible live price. A card-heavy client homepage, a dead-on-arrival sports tab, and speculative crypto chip rows all change visible token count, hydration volume, and subscription fan-out. Even if the UI "works," it leaves the realtime sessions inheriting accidental load characteristics rather than a deliberate baseline.
6. Theme bootstrapping is architecturally wrong, not just visually wrong.
   Hard-coding `data-theme="dark"` in the root layout prevents parity with live Polymarket, but more importantly it bakes theme selection into SSR markup in a way that will fight any future system-pref or persisted-override model. The right baseline is CSS/system-driven defaulting, not a fixed server markup choice.
7. The card boundary deserves another look.
   `EventCard` is currently client-only largely because its internal Yes/No buttons call `router.push`, even though `CardShell` already provides link navigation. That may still be acceptable, but architecturally it is the next obvious place to ask whether the page is hydrating interaction because it is needed, or because navigation semantics were implemented in the most convenient way.

### Homepage Shell

1. Keep the rogue homepage as-is because it already exists.
   Rejected. The section structure is useful, but the current dark chrome, oversized cards, chip wall, and local-only filters do not match the 2026-04-19 live product.
2. Revert all the way back to the original Session `06` bare card grid.
   Rejected. That would be simpler, but it leaves fidelity on the table because live Polymarket still uses editorial homepage modules above the main feed.
3. Keep the section architecture and refactor it toward the current live homepage.
   Adopted. This keeps the right product shape while removing the custom dashboard drift.

### Implementation Brainstorm

#### Server / Client Boundary Alternatives

1. Keep the current page-level client components and accept the extra hydration as the cost of richer UI.
   Rejected. This is the fastest path, but it blurs the architecture right before realtime work and makes the app harder to defend in review.
2. Move deterministic section building back to the server and keep only small client islands for controls that truly need local state.
   Adopted. This keeps RSC as the primary rendering model and limits hydration to the parts that earn it.
3. Push even more to the client and treat the app as an SPA over server-delivered data.
   Rejected. It works against the chosen stack and gives away one of the strongest architectural choices in the project.

#### Homepage Implementation Alternatives

1. Keep `HomePage` as one large client component with local filter state and derived sections.
   Rejected. Too much deterministic work in the browser.
2. Make the entire homepage server-rendered and convert every control into a route/search-param transition.
   Rejected. Architecturally pure, but too heavy-handed for lightweight on-page browsing controls.
3. Split the homepage into server-selected hero/breaking/topics modules plus a smaller client island only where interactive filtering is justified.
   Adopted. This is the cleanest balance of fidelity, performance, and implementation simplicity.

#### Crypto Implementation Alternatives

1. Preserve the current `CryptoExplorer` and just prune the chip rows.
   Rejected. The route entrypoint is still too client-heavy and still encodes unsupported assumptions.
2. Return to the old grouped-expander concept from the earlier plan.
   Rejected. It solves different assumptions, not the current product reality.
3. Build explicit server-side crypto facet derivation helpers, then render a list-first crypto surface with only data-backed client controls.
   Adopted. This turns parsing and grouping into a deliberate boundary instead of incidental route state.

#### Sports Implementation Alternatives

1. Keep the `Live | Futures` client tab model and just change the default tab.
   Rejected. It would fix the empty state symptom but keep the wrong architecture at the route entrypoint.
2. Revert to a server-filtered baseline category surface first, then layer a dedicated sports surface once grouping rules are re-validated.
   Adopted. This restores a safe architecture immediately and reduces surprise for the realtime sessions.
3. Introduce nested `/sports/live` and `/sports/futures` routes now.
   Rejected for now. It may become correct later, but it is premature before the feed shape and UX target stabilize.

#### Theme Implementation Alternatives

1. Force dark in SSR markup and revisit later.
   Rejected. Wrong parity and bad bootstrap model.
2. Force light in SSR markup and revisit later.
   Rejected. Better visually, still wrong architecturally.
3. Use system-pref-driven theming by default and reserve explicit overrides for a later, deliberate theme model.
   Adopted. This matches the live product and is the cleanest long-term bootstrap behavior.

#### Card / Grid Boundary Alternatives

1. Keep `EventGrid` and `EventCard` client-side.
   Rejected for `EventGrid`; not accepted as the default posture.
2. Restore `EventGrid` to the server immediately, and separately evaluate whether card navigation semantics can remove some of `EventCard`'s client-only surface.
   Adopted. This is the right immediate fix without forcing a larger card rewrite into the review session.
3. Rewrite the entire card stack now to eliminate `useRouter` and every card-level client dependency before continuing.
   Rejected for now. Worth exploring in `08A`, but too much churn to mandate from the review alone.
