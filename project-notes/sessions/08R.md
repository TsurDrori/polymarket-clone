## Key Findings

1. The rogue homepage kept the right broad modules, but not the right visual language. Current live Polymarket still has a homepage hero, breaking-news module, hot-topics module, and an all-markets feed. The local version is materially darker, denser, and more dashboard-like than the live product.
2. Theme baseline is wrong. A direct browser check on 2026-04-19 with forced `light` and `dark` color schemes showed that live Polymarket follows system `prefers-color-scheme` by default. The local app hard-codes a dark shell. For this clone, baseline parity means system-pref-driven theming rather than a forced single theme.
3. Politics, crypto, and sports diverged harder than the homepage. Current live category pages are list-first and much lighter. The landed local pages read like custom internal dashboards rather than Polymarket category surfaces.
4. The current sports route is functionally broken. On 2026-04-19, a live top-volume sports slice fetched from Gamma returned `25` visible events at `limit=30`, and the first `12` were all futures/outrights. The local route defaults to a `Live` tab built from game detection, so the initial view renders an empty state instead of markets.
5. The current crypto filters are not grounded in current live data. On the same top-30 crypto slice, the rogue helpers resolved time buckets to only `all: 13` and `pre-market: 17`, market type to `all: 30`, and lead-coin chips to `all: 24`, `bitcoin: 4`, `ethereum: 1`, `microstrategy: 1`. Most of the shipped chip rows are therefore decorative or dead.
6. `EventGrid` should not have been moved behind a client boundary. Session `06` intentionally kept it server-side so only `EventCard` and `PriceCell` client islands pay the hydration cost. The current `'use client'` on `EventGrid` buys nothing and makes the bundle boundary worse.
7. The current tag contract is mostly correct and should stay narrow. `isEventVisible()` hiding only `hide-from-new` remains the right event-suppression rule; category headline tags still arrive as `forceHide: true` and should not suppress whole events.

## Decisions

### Homepage Shell

Alternatives considered:

1. Keep the rogue homepage as-is because it already exists.
   Rejected. The section structure is useful, but the current dark chrome, oversized cards, chip wall, and local-only filters do not match the 2026-04-19 live product.
2. Revert all the way back to the original Session `06` bare card grid.
   Rejected. That would be simpler, but it leaves fidelity on the table because live Polymarket still uses editorial homepage modules above the main feed.
3. Keep the section architecture and refactor it toward the current live homepage.
   Adopted. This keeps the right product shape while removing the custom dashboard drift.

### Global Shell

Alternatives considered:

1. Keep the current dark header/footer chrome and treat it as a stylistic interpretation.
   Rejected. It is too far from current Polymarket and reads as custom brand work instead of fidelity work.
2. Revert to the original minimal Session `07` shell with only the 4 category tabs.
   Rejected. Live Polymarket has richer global chrome than that, and the recruiter-grade version should acknowledge it.
3. Keep the richer-shell direction, but refactor it into a lighter, tighter, live-like header/footer/category frame.
   Adopted. The concept survives; the current implementation does not.

### Crypto Direction

Alternatives considered:

1. Keep the current filter-heavy card hub.
   Rejected. It depends on filters that current live data does not actually support and keeps the surface card-heavy when the live page is list-first.
2. Return to the older grouped-row concept from the pre-rogue Session `11`.
   Rejected. It is also stale relative to the 2026-04-19 live crypto page and would overfit to an older assumption.
3. Adopt a hybrid, list-first crypto surface that keeps dedicated crypto treatment but only ships filters backed by observed data.
   Adopted. This is the most defensible compromise between fidelity and differentiated engineering judgment.

Direction chosen:

- [x] crypto direction — `adopt a hybrid and rewrite Session 11 around it`

### Sports Direction

Alternatives considered:

1. Keep the current live/futures sportsbook hub.
   Rejected. It ships an empty default state today and assumes a game-heavy feed that current top-volume sports data does not provide.
2. Return to a simpler baseline category surface first, then design a better custom sports page from verified data.
   Adopted. This fixes the broken default route immediately and keeps the next sports iteration honest.
3. Split live rows and futures cards into explicit nested routes right now.
   Rejected. It adds routing complexity before we have a data-backed reason for the split.

Direction chosen:

- [x] sports direction — `return to a simpler grid-first category page`

### Shared Architecture

Alternatives considered:

1. Leave `EventGrid` client-side and lower fetch limits further to keep the pages cheap.
   Rejected. Next 16 docs do not justify this boundary, and the fetch caps are already harming UX on sports.
2. Restore `EventGrid` as a server component, keep Hydrator/PriceCell as client islands, use surface-driven fetch caps, and retain the narrow tag contract.
   Adopted. This is still the cleanest recruiter-grade architecture.
3. Push more of the feed stack client-side and fetch on the browser to sidestep server concerns.
   Rejected. It moves the app away from the chosen RSC architecture for no clear gain.
